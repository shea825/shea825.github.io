Gitlab CI/CD Manual

CI/CD 基本概念

  • CI
    • 从存储库或生产环境中进行构建和自动化测试，以确保没有集成问题并及早发现问题
    • 开发人员在merge和pull请求后触发构建、测试和新代码验证的管道
  • CD
    • 可以通过更改自动推送到发布系统来随时将软件发布到生产环境中
  • Gitlab Runner
    • 一个处理构建的应用程序
    • 可以单独部署，并通过API与Gitlab CI/CD 一起使用

      Gitlab CI/CD 工作原理

  1. 将代码托管到 Git 存储库
  2. 在项目根目录创建 ci 文件.gitlab-ci.yml，在文件中指定构建、测试和部署脚本
  3. Gitlab 将检测到它并使用名为 Gitlab Runner 的工具运行脚本
  4. 脚本被分组为作业jobs，它们共同组成了一个管道pipelines

    Gitlab Runner 特点

  • 作业运行控制：同时执行多个作业
  • 作业运行环境：
    • 在本地
    • 使用 Docker 容器
    • 连接到远程 SSH 服务器
  • 支持Bash，Windows PowerShell
  • 允许自定义作业运行环境
  • 自动重新加载配置，无需重启
  • 易于安装，可作为Linux，macOs和Windows的服务
  • 类型：
    • 共享类型shared，运行整个平台项目的作业，需要admin权限
    • 项目组类型group，运行特定项目组下的所有项目作业
    • 特定项目类型specific，运行指定的项目作业
  • 状态：
    • 锁定状态locked：无法运行项目作业
    • 暂定状态paused：暂时不会接受新的作业
  • 执行器
    • shell
    • docker

Gitlab Pipeline

  • 流水线语法检测
    • CI Lint：对 ci 文件.gitlab-ci.yml的内容进行验证
  • yml脚本语法
    • stages：阶段，阶段之间按定义顺序串行执行，默认情况下前一阶段出错后面的阶段都不执行（默认skip）
    • job：任务，自动化执行的最小单位
      1. 本身名字可自定义，例如build-14.04-job和upgrade-subversion
      2. 下置属性stage用于指明本job所属的阶段，同一阶段的不同工作并行执行
    • image：镜像，指定一个基础Docker镜像作为基础运行环境
      1. 可在定义为全局变量，即所有job都用此镜像
      2. 也可在每个job下定义为局部变量，即每个job用各自制定的镜像
    • service：工作期间运行的另一个docker镜像，并 link 到image关键词所定义的docker镜像。在使用image镜像的同时将此镜像作为附加容器运行，会使运行更容易和快捷。例如运行某些数据库容器redis
    • script：脚本，具体指shell脚本，其中的内容会被gitlab-runner执行
      1. 无需克隆当前项目并切换分支，在流水线中，每一个的job的执行都会将项目下载并缓存这些流程,但应注意项目中所需的库镜像本地不一定有，可在shell脚本开头安装
      2. 推荐在项目文件夹下新建shell脚本，在该脚本下实现自动化功能，然后在script下执行该脚本即可（bash shellname.sh)
    • artifacts：制品，将流水线过程中的一些文件，文件夹，打包压缩，提供一个外链供人下载，另外还能缓存用于后续job
      • name：默认名称 artifacts
      • expose_as：在合并请求中公开制品文件
      • when：制品创建条件
        • on_success：作业成功时上传，默认选项
        • always
      • expire_in：制品保留时长
        • 默认为30天
      • reports
        • junit：指定存放单元测试报告的路径，此选项默认关闭，开启会加大资源占用
    • tags：指定运行对应 tags 的 Runner
    • when：指定作业的运行时间
      • on_success
      • on_failure
      • always
      • manual
      • delayed
        • start_in: '30'：延迟30s
    • timeout：作业设置超时时间，防止Shared Runner被项目占用过长时间
    • parallel：配置要并行运行的作业实例数
    • rules：构建规则，顺序地指定各类运行条件
      • if: '条件'
      • changes
      • exists
      • allow_failure
    • cache：存储编译项目所需的运行时依赖项，指定项目工作空间中需要在作业之间缓存的文件或目录
      • 有global cache和job cache之分
      • paths
      • key：缓存标记，key不同的缓存之间是独立的，建议以job或branch名为key来为作业和分支实现特定的缓存

        CI/CD Template 模板库

  • 作用：将一些安装包管理工具（例如maven、apt、pypi、nuget）通过一个作业模板和不同的构建命令来实现。通过定义模板流水线，让这些流水线可以直接在项目中使用，当遇到个性化项目时先创建.gitlab-ci.yml文件来引用模板文件，再针对具体项目实现个性化需求。
  • 流程：
    1. 创建一个 Git 仓库用于存放模板
    2. 创建一个 Template 目录存放所有的 Pipeline 的模板
    3. 创建一个 Jobs 目录存放所有的 Job 模板
  • 构建工具集成

Job keywords

全局关键词

关键字 是否必须 描述
image 否 用于docker镜像，查看docker文档
services 否 用于docker服务，查看docker文档
stages 否 定义构建阶段
before_script 否 定义在每个job之前运行的命令
after_script 否 定义在每个job之后运行的命令
variable 否 定义构建变量
cache 否 定义一组文件列表，可在后续运行中使用

  • stages

  • artifacts

    • 将流水线过程中的一些文件，文件夹，打包压缩，提供一个外链供人下载，另外还能在后续的job中缓存,通常下载格式为zip
    • 参数
      • paths 文件路径
      • exclude 排除的文件
      • name 制品名称
      • expose_as 在UI页面导出的名称
      • untracked 布尔类型，是否将git忽略的文件加到artifacts中
      • when 何时上传artifacts
        • on_success
        • on_failure
        • always
      • expire_in 过期时间默认30天
      • reports 收集测试报告

        既是全局也是任务关键词

  • image

    • 用于指定一个基础Docker镜像作为基础运行环境
    • 例如：c++ node java
    • 既可以设置当前流水线的，也可以设置当前任务的

  • cache

    • 用来指定需要在job之间缓存的文件或目录。只能使用该项目工作空间内的路径
    • 将当前工作环境目录中的一些文件，一些文件夹存储起来，用于在各个任务初始化的时候恢复。避免多个下载同样的包，能够大大优化流水线效率
    • 既可以设置当前流水线的，也可以设置当前任务的
    • 参数
      • paths 当前工作环境下的目录
