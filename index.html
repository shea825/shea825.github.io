什么是 Docker ?

  • Docker 是一个开源的应用容器引擎，基于 Go 语言 并遵从 Apache2.0 协议开源。

  • Docker 可以让开发者打包他们的应用以及依赖包到一个轻量级、可移植的容器中，然后发布到任何流行的 Linux 机器上，也可以实现虚拟化。

  • 容器是完全使用沙箱机制，相互之间不会有任何接口（类似 iPhone 的 app）,更重要的是容器性能开销极低。

网络应用程序常见的两种设计模式

  • C/S模式
    • 客户机位于目标主机可以保证性能，将数据缓存客户端本地从而提高数据传输效率
    • 采用协议相对灵活
    • 客户端和服务器都需要自己设计，开发量大
    • 对用户主机的安全性构成威胁
  • B/S模式
    • 没有独立的客户端，使用标准浏览器作为客户端，开发量小
    • 移植性非常好，不受平台限制
    • 只能选择http和https协议

Gitlab CI/CD Manual

CI/CD 基本概念

  • CI
    • 从存储库或生产环境中进行构建和自动化测试，以确保没有集成问题并及早发现问题
    • 开发人员在merge和pull请求后触发构建、测试和新代码验证的管道
  • CD
    • 可以通过更改自动推送到发布系统来随时将软件发布到生产环境中
  • Gitlab Runner
    • 一个处理构建的应用程序
    • 可以单独部署，并通过API与Gitlab CI/CD 一起使用

      Gitlab CI/CD 工作原理

  1. 将代码托管到 Git 存储库
  2. 在项目根目录创建 ci 文件.gitlab-ci.yml，在文件中指定构建、测试和部署脚本
  3. Gitlab 将检测到它并使用名为 Gitlab Runner 的工具运行脚本
  4. 脚本被分组为作业jobs，它们共同组成了一个管道pipelines

    Gitlab Runner 特点

  • 作业运行控制：同时执行多个作业
  • 作业运行环境：
    • 在本地
    • 使用 Docker 容器
    • 连接到远程 SSH 服务器
  • 支持Bash，Windows PowerShell
  • 允许自定义作业运行环境
  • 自动重新加载配置，无需重启
  • 易于安装，可作为Linux，macOs和Windows的服务
  • 类型：
    • 共享类型shared，运行整个平台项目的作业，需要admin权限
    • 项目组类型group，运行特定项目组下的所有项目作业
    • 特定项目类型specific，运行指定的项目作业
  • 状态：
    • 锁定状态locked：无法运行项目作业
    • 暂定状态paused：暂时不会接受新的作业
  • 执行器
    • shell
    • docker

Gitlab Pipeline

  • 流水线语法检测
    • CI Lint：对 ci 文件.gitlab-ci.yml的内容进行验证
  • yml脚本语法
    • stages：阶段，阶段之间按定义顺序串行执行，默认情况下前一阶段出错后面的阶段都不执行（默认skip）
    • job：任务，自动化执行的最小单位
      1. 本身名字可自定义，例如build-14.04-job和upgrade-subversion
      2. 下置属性stage用于指明本job所属的阶段，同一阶段的不同工作并行执行
    • image：镜像，指定一个基础Docker镜像作为基础运行环境
      1. 可在定义为全局变量，即所有job都用此镜像
      2. 也可在每个job下定义为局部变量，即每个job用各自制定的镜像
    • service：工作期间运行的另一个docker镜像，并 link 到image关键词所定义的docker镜像。在使用image镜像的同时将此镜像作为附加容器运行，会使运行更容易和快捷。例如运行某些数据库容器redis
    • script：脚本，具体指shell脚本，其中的内容会被gitlab-runner执行
      1. 无需克隆当前项目并切换分支，在流水线中，每一个的job的执行都会将项目下载并缓存这些流程,但应注意项目中所需的库镜像本地不一定有，可在shell脚本开头安装
      2. 推荐在项目文件夹下新建shell脚本，在该脚本下实现自动化功能，然后在script下执行该脚本即可（bash shellname.sh)
    • artifacts：制品，将流水线过程中的一些文件，文件夹，打包压缩，提供一个外链供人下载，另外还能缓存用于后续job
      • name：默认名称 artifacts
      • expose_as：在合并请求中公开制品文件
      • when：制品创建条件
        • on_success：作业成功时上传，默认选项
        • always
      • expire_in：制品保留时长
        • 默认为30天
      • reports
        • junit：指定存放单元测试报告的路径，此选项默认关闭，开启会加大资源占用
    • tags：指定运行对应 tags 的 Runner
    • when：指定作业的运行时间
      • on_success
      • on_failure
      • always
      • manual
      • delayed
        • start_in: '30'：延迟30s
    • timeout：作业设置超时时间，防止Shared Runner被项目占用过长时间
    • parallel：配置要并行运行的作业实例数
    • rules：构建规则，顺序地指定各类运行条件
      • if: '条件'
      • changes
      • exists
      • allow_failure
    • cache：存储编译项目所需的运行时依赖项，指定项目工作空间中需要在作业之间缓存的文件或目录
      • 有global cache和job cache之分
      • paths
      • key：缓存标记，key不同的缓存之间是独立的，建议以job或branch名为key来为作业和分支实现特定的缓存

        CI/CD Template 模板库

  • 作用：将一些安装包管理工具（例如maven、apt、pypi、nuget）通过一个作业模板和不同的构建命令来实现。通过定义模板流水线，让这些流水线可以直接在项目中使用，当遇到个性化项目时先创建.gitlab-ci.yml文件来引用模板文件，再针对具体项目实现个性化需求。
  • 流程：
    1. 创建一个 Git 仓库用于存放模板
    2. 创建一个 Template 目录存放所有的 Pipeline 的模板
    3. 创建一个 Jobs 目录存放所有的 Job 模板

文件的基本操作

open函数

  • flags取值 (可用|组合)
    • 文件打开模式
      • O_RDONLY = 0
      • O_WRONLY = 1
      • O_RDWR = 2
    • 非文件打开模式
      • O_APPEND 每次写的数据放在文件尾部
      • O_CREART
      • O_EXCL 配合O_CREART来创建文件，否则出错
  • 第三个参数取值
    多用户访问权限取值
  • fd取值
    • 文件存在时返回文件描述符
    • 通常Linux的API函数出错返回-1
      • 出错处理

lseek函数

  • offset参数
    • 相对偏移量
    • 有符号数类型
类型 32位平台 64位平台
char 1个字节8位 1个字节
short 2个字节16位 2个字节
int 4个字节32位 4个字节
long 4个字节 8个字节（区别）
long long 8个字节 8个字节
point 4个字节 8个字节（区别）
  • whence参数
    • 和offset配合使用
    • whence取值
      • SEEK_SET 偏移量 = offset
      • SEEK_CUR 偏移量 = 当前文件偏移量 + offset
      • SEEK_END 偏移量 = 文件长度 + offset
  • f_offset取值
    • 函数调用成功返回文件偏移量
    • 失败返回-1

read和write函数

  • buf参数
    • 指向用于保存所读出数据的缓存
    • write函数不会对buf所指向的内存有任何改动，因此加上const
  • count参数
    • 指示了需要从文件读或写出多少字节
  • rd_bytes取值
    • rd_bytes = 0 代表已读到文件末尾
  • 关于write的两个问题
    • 在文件中间某处写入数据是覆盖该处的数据
    • 当以O_APPEND标志打开文件时，可以使用lseek修改文件偏移量，但每次写都会把数据写到文件末尾
    • fd = 0 代表键盘的标准输入的文件描述符
      fd = 1 代表屏幕的标准输出的文件描述符
      scanf下层调用的就是read，read自动使用fd = 0来读数据

close函数

  • cl_tag取值
    • cl_tag = 0 函数调用成功
  • 是否存在内存泄漏？
    • 定义 分配了内存，想释放却没办法
    • open打开或创建一个文件后，若未close，其fd还存在，随时可调用close关闭，不存在内存泄漏

无信息搜索/盲目搜索

广度优先搜索BFS

特点

  1. 完备的 优先找到最浅的目标节点 但最浅不一定最优
  2. 如果路径耗散是节点深度的非递减函数，BFS才是最优的
    例如单步耗散相等的情况
  3. 内存的需求是比执行时间的消耗更大的问题

代价一致搜索UCS

特点

  1. 扩展路径消耗最低的节点
  2. 扩展到一个具有能返回同一状态的零耗散行动的节点时会陷入无限循环

深度优先搜索DFS

特点

  1. 对内存需求很少，只存储一条从根节点到叶节点的路径，以及该路径上每个节点的所有未被扩展的兄弟节点
  2. 另一种变形回溯搜索所用内存更少，每次只产生一个后继节点
  3. 有可能错误地选择一条分支一直走下去

GAMA入门小结

Posted on 2021-08-26 Edited on 2021-11-28

学习报告

问题

  • 情景
    送客到机场的出租车如果前往到达区排队等待载客返回市区，出租车必须到指定的“蓄车池”排队等候，依“先来后到”排队进场载客，等待时间长短取决于排队出租车和乘客数量的多少。

  • 问题内容
    某机场“乘车区”现有两条并行车道，管理部门应如何设置“上车点”，并合理安排出租车和乘客，在保证车辆和乘客安全的条件下，使得总的乘车效率最高。

    怎么用多智能体的模型进行动态编程？

  • 首先要了解多智能体系统的特征，它是由一个在一个环境中交互的多个智能体组成的计算系统，因此系统中各智能体之间存在数据和行为的交互，另外，如果把环境也当作一个智能体的话，各智能体也需要随时对外界环境做出反应，环境也会随着智能体而发生改变。

  • 其次要对多智能体模型进行抽象，使之符合面向对象的思想。一个智能体可以向其他智能体发送信息，这是数据的交互，在编程中具体表现为各智能体间局部变量的赋予；智能体也可以对其他智能体执行特定的操作，而被执行者需要做出反应，这是行为的交互，在编程中则表现为各智能体间函数的衔接或者迁移。我认为衔接类似于操作系统中进程的同步和互斥，但这种同步和互斥应由各个智能体自己独立完成，迁移则是智能体在运动过程中的模仿和学习。

  • 最后针对具体平台进行实现。

    怎么用GAMA实现这个系统？

  • GAMA是一个基于agent的建模和仿真开发环境，它的特点是仿真可空间显式，有可交互的用户界面。

  • 我针对机场出租车问题，定义了三个species，family代表乘客，taxi代表出租车，road代表道路。该问题主要探讨上车点在哪儿设置以及出租车和乘客如何安排，因此必须对出租车设置一种规则，对taxi分批放行，让乘客遵守秩序，family排队候车。

  • 在各species中我定义了一些信息，以taxi为例，包括速度、反应时间、自身大小和行走路径等等，除信息外也需要定义行为，仍以taxi为例，包括行驶、等待、接客和减速等。

  • 重点考虑各agent之间的交互问题，例如乘客移动时考虑简单的碰撞，当前方有乘客时懂得降速或者停等，乘客到达上车点时询问对应车辆的状态，车辆之间距离不得过近。

  • 不考虑同乘一辆车的乘客间的行为，当作一个整体，设为species family，其中有单人、双人、三人和四人之分，他们大小有别，速度在一个正常区间内浮动。另外，考虑到人的反应时间，重新设置全局变量step <- 0.2#s。

  • taxi分批定量放行，一次放行数量和方案有关，当一批车全部离开后放行下一批车

    • 上方道用于离开，下方道用于放行接客
    • 两条道都用于接客和离开，但为了安全应设置一定规则

  • family在视框最下面一个区域内随机出现，然后寻找上车点，在自己选择的上车点附近排队。排队遵循先来先服务原则，当family到达上车点时，设置局部变量ready_to_get_on <- true；当taxi到达上车点，查看ask该等待队列中是否有ready_to_get_on <- true的family，如果有则建立联系。然后taxi开始接family，设置一个准备时间用于模拟乘客放行李和上车的过程，该时间过后出租车离开场景。

    我学到了什么？

  • 虽然多智能体通常能够合作解决任何单个成员无法解决的问题，但使用多智能体系统解决现实问题是复杂的，需要我们定义智能体和环境并实现智能体之间的交互行为，而采用何种方式来实现这种交互是需要重点考虑的问题，使用过于简单的方法或者功能会让系统不够真实，使用算法搜索或者强化学习之类的方式来让agent变得智能有时候又很难去实现。

  • 我发现GAMA平台上支持GIS数据和BDI模型，如果导入实际机场的shapefile模型，通过设计智能体行为而让它们具有社会关系、情感和个性，并添加社会规范、义务和强制等规则，那么整个系统会变得更加真实，这也是我今后努力的方向。

    为什么多智能体系统能够解决复杂动态问题？

  • 智能体可以看作计算机程序，它们在开放和分布式环境中代表用户自主行动，以解决越来越多的复杂问题。一个用于解决复杂问题的计算机程序可以由多个子程序同时执行来完成，如果用多个智能体能达到同样的效果，我们就可以用多智能体系统来解决这个问题。多智能体系统是一个松散耦合的软件智能体网络，它们相互作用以解决超出每个问题解决者个人能力或知识范围的问题。相比于使用单一智能体，多智能体则具有较高的健壮性以及可拓展性，这使得其对于复杂环境具有较高的抗干扰能力。而相比使用集中式系统，由于多智能体系统分散的特性，不会遭受与集中式系统相关的诸如资源限制、性能瓶颈或单点故障的困扰。

  • 基于多智能体的仿真模拟是一种建模系统的方法，这些系统是分布式的，并且结合了博弈论、复杂系统、紧急情况、计算社会学、多主体系统和进化规划等许多元素，涉及到人与机器之间的复杂交互，具有真实性和科学性。智能体被视为一种计算模型，允许模拟环境中自主个体的行动和交互，也允许他们在整个系统中各自产生影响，因此是一种解决任务分配、团队规划、用户偏好、开放环境等的更自然的方式。
